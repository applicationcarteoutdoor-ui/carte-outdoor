## Projet

**SpotMap** (renommée en v62, ex-« Carte Outdoor » — le dépôt, l'URL GitHub Pages et les clés localStorage `carte-outdoor:*` NE CHANGENT PAS) : PWA statique de cartographie de points d'intérêt outdoor (via ferrata, refuges, escalade, canyons, châteaux, villages classés, randonnées, GR, musées), **multi-pays (10) : France, Suisse, Italie, Espagne, Portugal, Allemagne, Pays-Bas, Luxembourg, Belgique, Nouvelle-Zélande** (page de garde = **vraie carte du monde cliquable** — tuiles OSM, zoom/déplacement LIBRES (minZoom 1), pays disponibles surlignés en teal semi-transparent + points verts pulsants, pays « bientôt » = points ambrés (liste `PAYS_BIENTOT` dans app.js), **nom du pays au survol** (tooltips ; `Intl.DisplayNames` fr sur `data/monde.geojson`, Natural Earth domaine public sans Antarctique — les polygones des autres pays sont invisibles mais survolables) ; cf. `js/config/pays.js` et `choisirPays()`). Logo : spot + « ? » (`icons/icon.svg`, PNG régénérés par `tools/generer_icones.py`). **Filtres VIERGES à chaque activation de catégorie** (le cochage d'une catégorie fait `delete state.filterSelections[id]` — demande utilisateur v64). **PACKS de catégories (v72+)** : la sidebar range les catégories en packs (`js/config/packs.js`, drill-down 3 vues) — voir la section Architecture. **PROFIL + badges 100 % local (v77)** : `js/profil.js` + `js/config/badges.js`, tuile 👤 dans Réglages. **GROSSES catégories EXCLUSIVES (v66)** : les couches lourdes (**France : toilettes/eau/grottes** ; les MUSÉES ont quitté le rang des couches lourdes en v76 → fusionnés dans `points.geojson`, catégorie normale sans avertissement, disponible hors ligne) s'affichent SEULES — en activer une décoche tout, cocher une petite décoche la grosse, seules les petites se combinent (logique dans `onToggleTheme` + assainissement des prefs héritées dans `restaurerPrefs`). **Vanilla JS + modules ES, aucun build, aucune dépendance npm** — les librairies (Leaflet, markercluster, togeojson, fflate) sont vendorées dans `vendor/`. Interface, commentaires et documentation en **français**. **RÈGLE QUALITÉ (v75)** : chaque point VISE 1 lien + 1 photo + 1 description (manques tolérés, mais plus de catégorie « nue » — lien « 🔎 Infos » de repli systématique, description factuelle composée des faits). ~25 900 points France par défaut, dont ~5 050 musées (fusionnés depuis v76), 4 375 panoramas, 3 351 sites d'escalade, 3 020 arbres remarquables, 1 274 canyons, 579 cols mythiques… + 190 tracés GR + des couches volumineuses en fichiers séparés chargés à la demande (non pré-cachés) : France ~66 500 toilettes (`data/toilettes.geojson`), ~49 500 points d'eau (`data/eau.geojson` : fontaines, robinets, sources, avec filtre de potabilité) et ~49 700 grottes/cavités de spéléologie (`data/grottes.geojson` : socle Grottocenter ODbL, 6 filtres) ; **depuis la v67, toilettes + eau existent pour TOUS les pays** (`data/<iso>/toilettes.geojson` + `eau.geojson` : IT 10 211/79 138, ES 7 404/60 144, CH 6 588/18 023, NZ 4 562/1 957 ; v69 : DE 30 939/38 661, PT 3 898/8 042, NL 3 196/3 493, BE 2 382/2 129, LU 333/623). Trois fonctions connectées : **l'Oracle** 🔮 (« quoi faire autour d'un code postal ou d'une commune » — mode gratuit sans clé, ou IA avec la clé Anthropic/Google de l'utilisateur), la **sync multi-appareils** (Supabase + compte Google) et la **fréquentation anonyme** 📊 (`js/frequentation.js` + `supabase/frequentation-schema.sql`, guide `docs/FREQUENTATION.md` : ping de présence UUID aléatoire toutes les 4 min, totaux « en ce moment / aujourd'hui / 7 jours » dans ⚙️ Réglages ; table invisible par RLS, seuls des agrégats sortent).

## Règles critiques

- **RÈGLE QUALITÉ (v75, demande utilisateur)** : chaque point livré doit VISER **1 lien + 1 photo + 1 description** — les manques restent tolérés, mais on ne livre plus de catégorie « nue ». Minimum systématique : un lien « 🔎 Infos » (recherche web) quand rien d'autre n'existe ; description FACTUELLE composée des faits (altitude, espèce, hauteur…) plutôt que rien — jamais de prose inventée. Photos/articles par géo-recherche Wikipédia (`tools/enrichir_geosearch_pays.py`, garde anti-homonyme : mot commun ≥ 4 lettres OU < 150 m) — pour les pays construits par `construire_pays.py`, les résultats vivent dans `tools/geosearch-<iso>.json` (rejoués au build, JAMAIS d'édition directe des data/<iso>). Tout NOUVEAU pipeline de catégorie doit appliquer cette règle dès sa première récolte.

- **Incrémenter `VERSION` dans `sw.js` après toute modification de fichiers de l'app** — mais seulement quand l'ensemble est finalisé : une version bumpée en cours de chantier fige un état intermédiaire dans le cache des visiteurs (déjà causé des « bugs » fantômes). Depuis la v9, `app.js` recharge automatiquement la page au `controllerchange` : la nouvelle version s'applique au premier chargement. Depuis la v31, `app.js` appelle aussi `reg.update()` au retour au premier plan (`visibilitychange`) et toutes les heures → une PWA installée restée ouverte se met à jour seule.
- **Pré-cache du shell avec `fetch(new Request(url, { cache: "reload" }))`, jamais `cache.addAll()`** : `addAll` passe par le cache HTTP du navigateur, donc un fichier dont le CONTENU change sans que le NOM change (ex. `img/carnet-*.jpg` remplacées) était recopié PÉRIMÉ dans le nouveau cache — la VERSION montait mais les pixels restaient anciens (bug « couverture avec le bureau » v24→v26, plusieurs versions à diagnostiquer).
- **Jamais de `confirm()`/`alert()` natifs** : silencieusement ignorés dans certains contextes (PWA installée, dialogues bloqués) — utiliser `confirmer()` de `js/import-export.js` (dialogue intégré).
- **Échappement : uniquement `esc()` de `js/util.js`** (échappe aussi `"` et `'`, donc sûr dans les attributs). Jamais d'échappement local ni l'astuce `div.textContent` : elle n'échappe pas les guillemets → injection d'attribut possible depuis un fichier de sauvegarde importé (noms de points, libellés de catégories = données tierces).
- **CSP** : la balise meta de `index.html` liste les origines réseau autorisées — tout nouvel appel externe (fournisseur IA, back-end…) exige d'ajouter son domaine à `connect-src`, sinon blocage silencieux. Même piège pour les **images** : `img-src` doit lister `https://upload.wikimedia.org` (les photos des châteaux/cathédrales/cités viennent de Wikipédia — leur oubli les a fait disparaître de la v35 à la v41, sans aucune erreur visible).
- **Les clés API de l'Oracle sont secrètes** : localStorage de l'appareil uniquement, JAMAIS dans l'export de sauvegarde ni dans la sync. À l'inverse, l'URL et la clé `anon` Supabase (`js/config/supabase.js`) sont publiques par conception (RLS protège les données).
- **Ne jamais conseiller « vider les données du site »** à l'utilisateur : cela efface son carnet et ses points (IndexedDB) — et c'est inutile pour recevoir une mise à jour depuis les correctifs du SW.
- En dev, le service worker sert son pré-cache : pour tester une modification, désinscrire le SW + vider les caches dans le navigateur (ou mode navigation privée). `dev_server.py` envoie `no-store` mais ne neutralise pas le SW.
- Les **ids de catégorie** (`themes.js`) sont stables et stockés dans les données des points : ne jamais les renommer. Seul le `label` est modifiable. **Ids de points aussi stables** (statuts/carnet des utilisateurs pointent dessus) : jamais renumérotés ; une revue de données recale/enrichit EN PLACE, les ajouts prolongent la séquence.
- **Licences des données** : les **FAITS** (coordonnées, cotations, hauteurs, comptes) ne sont pas protégeables et sont réutilisables ; les **descriptions rédigées et les photos** relèvent du droit d'auteur/licences (CC-BY-SA de Camp to Camp, etc.) — **ne pas les recopier**, s'en tenir aux faits + un lien vers la source. Attribution requise pour Licence Ouverte/ODbL : section « Sources des données » dans le dialogue ⚙️ Réglages (`index.html`). Photos affichables : `upload.wikimedia.org` uniquement (CSP).
- `dev_server.py` doit rester sur `ThreadingHTTPServer` (un serveur mono-thread bloque le chargement : le navigateur ouvre des connexions parallèles).

## Architecture

**Multi-pays (v60)** : l'app couvre plusieurs pays — **`js/config/pays.js`** (SHELL) est le registre : par pays, fichier de points (`data/points.geojson` FR, `data/nz/points.geojson` NZ — **seul celui de la France est pré-caché**), surcouche de grands itinéraires (`gr.fichier/label/compte/fiche` : Sentiers GR ↔ Great Walks), catégories disponibles (`categories` = liste ou null=toutes, `categoriesExclues` — les catégories perso et celles avec points utilisateur restent toujours visibles), `couchesLourdes` (**table `id → fichier` depuis la v67** : toilettes + eau pour tous les pays, grottes/musées en plus pour la France ; le FAB 🚻 ne se masque que si `toilettes` en est absent), vue de départ et `wikiLang` (Oracle). Le pays choisi vit dans `carte-outdoor:pays` (localStorage, lecture SYNCHRONE — map.js/sidebar.js l'importent). **Page de garde** `#pays-overlay` au premier boot (`choisirPays()` dans app.js), bouton « 🌍 Changer de pays » dans ⚙️ Réglages (changement → `location.reload()`). Vue mémorisée **par pays** (prefs `vue_fr`/`vue_nz`). Au boot, les catégories actives inconnues du pays sont décochées (sinon leurs filtres polluent le panneau) ; si plus rien n'est coché, la 1re catégorie du pays est activée. **Le carnet et les statuts sont COMMUNS à tous les pays** (ids de points préfixés par pays : `nz-hut-0001`) : à l'ouverture du carnet, `chargerPointsAutresPays()` charge les points des autres pays (hook `avantOuverture` de carnet.js) pour résoudre tous les noms. Ajouter un pays = une entrée dans `PAYS` + `data/<pays>/points.geojson` (+ éventuelle surcouche d'itinéraires).

`js/app.js` est le seul propriétaire de l'état partagé (`state` : points, statuts, catégories cochées, sélections de filtres) et câble tous les modules par callbacks à l'init. Les modules ne s'importent presque pas entre eux ; le flux est : interaction → callback vers app.js → mutation de `state` → `rafraichir()` (re-rend panneau gauche + filtres + marqueurs + persiste les prefs). Les prefs sont écrites avec un debounce de 400 ms (`planifierPrefs`), vidé sur `visibilitychange`/`pagehide`. Au boot, un garde intercepte les retours OAuth : si `window.name === "carte-oauth-popup"` avec un jeton dans le hash, la page appelle `finaliserPopupAuth()` au lieu de démarrer l'app.

- **`js/config/packs.js` (v72, contenus retravaillés v73-v76)** — les PACKS de catégories, rangement de la sidebar : trois couches MIROIR de themes.js (`PACKS` par défaut — **montagne/nature(=« Eau »)/paysages/culture/road-trip(=« Services »)**, ids stables, contenus DISJOINTS pour les packs par défaut (une catégorie = un seul pack ; les packs perso mélangent librement) —, `customPacks` via `registerCustomPacks`, `packOverrides` via `setPackOverrides` — **les overrides de packs incluent `categories`** pour réordonner un pack d'usine, « Réinitialiser » restaure). `getPack()/allPacks()/isCustomPack()/packExists()/packsDeCategorie()`. Pack VIRTUEL `mes-categories` (perso + points utilisateur + « autre », calculé au rendu). La sidebar (js/sidebar.js) navigue en DRILL-DOWN 3 vues : accueil tuiles `.pack-grid` — **CLIC sur une tuile = tout afficher/masquer le pack** (`onTogglePack`, les couches lourdes restent individuelles), **▸ = entrer dans le pack** (+ Suivi/Tracés + « ☰ Toutes les catégories » + « ＋ Créer un pack ») —, pack ouvert (`ouvrirPack(id)`/`retourAccueilPacks()` exportées — le tuto s'en sert ; **fermer la sidebar ramène TOUJOURS à l'accueil des packs**, v76), liste complète (rendu historique). Écran **Créer un pack** refondu (v76 : aperçu live, catégories en pastilles `.pack-chip`). Pref `sidebarVue {mode, packOuvert}` (validée par `packExists` au boot), customPacks/packOverrides dans storage.js (`carte-outdoor:customPacks`/`packOverrides`) ET la sync. Les cases passent par l'`onToggleTheme` existant : exclusivité couches lourdes/filtres INTACTS. **Filtres discrets (v76)** : le panneau ne s'ouvre plus tout seul (`filtersCollapsed` défaut true, bouton en haut-droit). **Le HASH d'URL est RÉSERVÉ aux liens partageables** : `routerHash()` (app.js) traite `#pack=<id>`/`#cat=<id>` (jamais une couche lourde), `#pt=` viendra — ne JAMAIS utiliser le hash pour autre chose. Ids réservés (éternels) : packs sport/gastronomie/loisirs/professionnel.
- **`js/profil.js` + `js/config/badges.js` (v77)** — PROFIL utilisateur + badges **100 % LOCAL** (rien sur un serveur) : tuile 👤 dans ⚙️ Réglages (`#profil-dialog`). Stats calculées depuis statuts « ✓ Fait » + sorties + traces GPX (chaque track porte déjà `.stats`), TOUS pays (résolveur = `chargerPointsAutresPays` comme le carnet) : lieux visités, sorties, pays, catégories, km + D+. **16 badges** à jauge (`BADGES` : `metrique` = clé de stats ou `theme:x+y` pour compter par catégorie ; `valeurMetrique()`), rétrospective annuelle. Pseudo + avatar (emoji cyclable) en prefs (suivent la sync). **PIÈGE : le pays d'un point se déduit de son id via `^(iso)-[a-z]{2,}-\d`** — un château FR est `ch-0002` (ch=château, PAS la Suisse) ; sans le format `<iso>-<abr>-<num>` la France compterait double.
- **Catégories « paysages » France (v72/v74/v75)** — scripts `tools/recolter_categories_v72.py` puis `_v74.py` (append dans points.geojson, ids stables croix-/col-/pano-/phare-/arbre-/ciel-####, relance = mise à jour EN PLACE par clé theme|nom|lat3) : `sommet-croix` (143, OSM `summit:cross=yes`), `col-mythique` (579 — cols OSM nommés gardés SEULEMENT si article Wikipédia fr au nom exact ≤ 10 km : la notoriété fait le « mythique » — filtre spécifique à la France), `panorama` (4 375, `tourism=viewpoint` nommés), `arbre-remarquable` (3 020, `natural=tree`+`denotation=natural_monument`), `phare` (338, `man_made=lighthouse`), `plongee` (362, `sport=scuba_diving`), `ciel-etoile` (**105** : 7 RICE DarkSky éditoriales + **98 observatoires** `man_made=observatory`, v75 — de VRAIS spots d'observation). **`village-abandonne` a été SUPPRIMÉ en v76** (thème + points retirés). Ces catégories sont rangées dans les packs 🌄 Paysages / 💧 Eau (`ciel-etoile` dans aucun pack, visible via « Toutes les catégories »).
- **`js/config/themes.js`** — LE point de configuration : catégories (id stable, label, couleurs, icône), champs de fiche, et **filtres déclaratifs** (types `tokens`/`prefix`/`contains`/`value`/`bucket`, évalués par `passeFiltre()` de `js/filtrage.js` — fonction pure, couverte par `dev/tests.js` ; les buckets lisent les champs numériques suffixés `_n` des `details`). Trois couches de catégories : `THEMES` (base), `customThemes` (créées par l'utilisateur, enregistrées via `registerCustomThemes()` au boot, supprimables), `overrides` (personnalisation label/couleurs/icône, appliquée par `getTheme()`). Toujours passer par `getTheme()`/`allThemes()` pour l'affichage. Catégories récentes : `randonnee` (🥾, filtres par effort Durée/D+/Distance en buckets lisant `duree_n`/`denivele_n`/`distance_n` ; tracé au clic), `lac` (🏞️, filtres Fiche + Altitude en bucket), `cascade` et `chateau` (filtre Fiche). La catégorie `cite-caractere` porte le label unifié **« Village »** avec un filtre `contains` sur `label` (⭐ Plus beau village de France / 🏘️ Cité de caractère) — les deux anciennes catégories ont été fusionnées en v46 sous cet id stable.
- **Couches volumineuses (`toilettes`, `eau`, `grotte`)** — modèle commun dans app.js, centralisé par la table `COUCHES_LOURDES` (textes UI) : catégorie normale de `themes.js` mais dont les points vivent dans un fichier séparé NON pré-caché, chargé à la première activation (`chargerCouche(id)` générique → fusionné dans `state.allPoints` par `chargerPoints`), rechargé au boot si la catégorie était déjà cochée. **Le FICHIER dépend du pays (v67)** : `pays.couchesLourdes` est une table `id → fichier` (France : toilettes/eau/grotte — **les musées ne sont PLUS une couche lourde depuis v76**, fusionnés dans `points.geojson` ; CH/IT/ES/… : toilettes + eau seulement, `data/<iso>/toilettes.geojson` + `eau.geojson`, récoltés par `tools/recolter_sanitaires_pays.py` — Overpass area ISO + tuiles subdivisées, ids stables dérivés des ids OSM `<iso>-wc-n…`/`<iso>-eau-n…` ; la NZ se récolte SANS bbox, aire seule). `coucheLourde(id)` renvoie la config seulement si le pays courant liste l'id — ailleurs la même catégorie est ordinaire (`grotte` en NZ/IT…). Le compteur affiche « … » tant que le fichier n'est pas chargé. À l'activation, un **dialogue d'avertissement** (`demanderModeCouche(dialogId)` sur `#wc-dialog`/`#eau-dialog`, boutons `.mode-retour/.mode-proximite/.mode-valider`) propose trois choix : Retour, « À moins de 1 km » (géoloc + zoom sur la position via `coucheAutourDeMoi(id)`, n'affiche que cette couche autour de soi), ou Valider (affichage complet). Ajouter une couche lourde = une entrée dans `COUCHES_LOURDES` + un `<dialog>` jumeau + un getter `pointsX`/`chargerX` + un compteur « … ». `grotte` : socle **Grottocenter** (API publique `api.grottocenter.org/api/v1/geoloc/entrances?sw_lat&…` — tout renvoyé par bbox en un appel ; FAITS ODbL seulement, jamais les topos/descriptions/photos protégés), `tools/recolter_grottes_grottocenter.py` + `tools/construire_grottes.py` (filtre France par région + géocodage inverse — la bbox mord sur l'Italie/Espagne ; type/progression/eau déduits du NOM ; fusion des 484 grottes Wikipédia `grot-XXXX` conservées + nouvelles `grot-g{id}` stables). Nombre de cordes / hauteur de puits = **impossibles en open data** (topos protégés). `eau` : récolté par `tools/recolter_eau.py` (Overpass `amenity=drinking_water`/`man_made=water_tap`/`natural=spring`…), `tools/normaliser_eau.py` pose `details.type` et `details.potabilite` en libellés lisibles (servant à la fois d'affichage et de valeur de filtre — la potabilité d'une source est « non garantie », jamais présumée). **Ne pas ajouter ces fichiers au SHELL de `sw.js`** (volumineux, mis en cache à la volée par le fetch handler).
- **`js/storage.js`** — toute la persistance, API volontairement async pour pouvoir brancher un back-end plus tard sans toucher au reste. localStorage (prefs, statuts, overrides, customThemes, idées) ; IndexedDB v3 (`DB_VERSION = 3`, stores : `userPoints`, `tracks`, `journal` = carnet par point avec photos, `carnet` = thème + images du grimoire dont le logo). Protection anti-effacement : `app.js` demande `navigator.storage.persist()` et rappelle une sauvegarde (export) au bout de 14 j sans export ; `dernierExport`/`dernierRappelSauvegarde` sont des prefs.
- **`js/map.js`** — performance : les marqueurs sont créés une fois et mis en cache (`cache: Map<id>`), les instances de divIcon sont partagées (une par combinaison thème/statut/sélection), l'icône n'est régénérée que si statut/thème/sélection change ; la **sélection fait partie de l'icône** (classe `pin-selected`) pour survivre aux reconstructions du clustering. Au-delà de 4 000 points, `setPoints` ajoute par tranches **triées par distance au centre** (la zone regardée s'affiche en ~1 s, le reste en fond) — **ne pas revenir au `chunkedLoading` de markercluster : il n'affiche RIEN avant la fin du lot complet** (~25 s pour 66 000 points). ## Pipeline de données (`tools/`)

`build_data.py` + `enrichissements.py` génèrent `data/points.geojson` et `data/gr.geojson` depuis `Donné/` et des sources en ligne. **Tout est mis en cache dans `tools/`** (`communes.json`, `refuges-api.json`, `wiki-cache.json`, `alti-cache.json`, `viaferrata-liste.html`, `grgo-index.html`) : relancer le script ne re-télécharge que le manquant.

- Refuges : API refuges.info (`api/bbox…detail=complet`) — le KML de `Donné/Refuge` est obsolète.
- Escalade/châteaux/cités : géocodés à l'origine au **centroïde de commune** (référentiel geo.api.gouv.fr, rapprochement exact → préfixe → contient, St↔Saint normalisé) — d'où la nécessité des **revues de données** (ci-dessous) pour recaler sur les vraies positions.
- Wikipédia : **ne pas utiliser l'endpoint de recherche** (rate-limité en rafale et renvoie l'article de la commune) — construire des titres candidats (« Château de X »…) et les vérifier par lots de 50 via `action=query&titles=`, en sauvegardant le cache à chaque tranche (les 429 arrivent par vagues, le retry attend 15-180 s). Les châteaux sans page trouvée restent dans `chateau` avec `details.fiche = "À vérifier"` (l'ancienne catégorie séparée `chateau-a-verifier` a été fusionnée en v45 — le filtre déclaratif « Fiche » ✅ Référencé / 🔍 À vérifier la remplace ; même convention pour les cascades et toute nouvelle catégorie).
- Grottes et cathédrales : récolte des arbres de catégories Wikipédia (`recolter_categorie_wikipedia`, sous-catégories suivies seulement si leur nom contient le mot-clé, sinon le parcours dérive hors sujet) puis `prop=coordinates|pageimages` — **`colimit=max` et suivi de `continue` obligatoires**, sinon l'API ne renvoie que 10 coordonnées par réponse (bug déjà rencontré : 80 % des points silencieusement sans coordonnées).
- D+ des GR : service altimétrique IGN (`data.geopf.fr/altimetrie`, POST par lots de 180 points, échantillonnage ~300 m) — c'est une estimation, affichée comme telle.
- Cascades / lacs / villages : récoltés par l'agent enrichisseur (scripts `recolter_cascades.py`, `recolter_lacs.py`, `recolter_villages.py`), avec **sélection qualitative** (les lacs anonymes ou minuscules sont écartés — 26 % de rejet) et convention `details.fiche`. Lacs recentrés dans l'eau, altitude corrigée par l'IGN.
- Randonnées : `recolter_randonnees.py` (points de départ + sommets, champ `massif`) puis `recolter_traces_randos.py` calcule le vrai **tracé sur le réseau de sentiers OSM** (routage Dijkstra départ→sommet, via ferrata/privé exclus, simplification Douglas-Peucker) → `data/randos.geojson`. Ensuite **`completer_stats_randos.py` est OBLIGATOIRE** : distance mesurée sur le tracé (détection aller/boucle), dénivelé IGN si manquant, durée estimée (4 km/h + 300 m/h). Il pose les champs numériques `distance_n`/`denivele_n`/`duree_n` qui alimentent les filtres par effort de la fiche randonnée. Chaque fiche porte 3 liens externes (`properties.links` : 🥾 Komoot, ⛰️ Altitude Rando, 🗺️ AllTrails). **Couverture France livrée : 135 randonnées, 28 massifs** (listes éditoriales `tools/randos_liste_*.py` + `recolter_randonnees_france.py` ; cache des tracés shardé `tools/traces-cache/`). Limite honnête : le tracé est le plus direct sur sentiers balisés, pas les variantes en boucle éditoriales.

Overpass rate-limite fort (429/504) : retries longs, cache sauvegardé seulement si toutes les tuiles réussissent. ## Pièges connus de l'environnement de preview

**Si le panneau de preview intégré est indisponible** (arrive selon les sessions) : piloter Chrome headless via CDP comme `dev/video/capturer.py` (classe `Cdp` réutilisable, `Runtime.evaluate` avec `awaitPromise`/`returnByValue`, `Emulation.setDeviceMetricsOverride` pour le mobile) sur le serveur de dev. Toujours **purger SW + caches + `localStorage`** avant de tester (sinon ancienne version / prefs polluées) ; les catégories de la sidebar sont des `.cat-row` (pas des `<li>`) ; les transitions CSS restent parfois figées en émulation mobile (tester avec `transition:none` avant de crier au bug). Le proxy du panneau de preview cache agressivement : si des fichiers périmés sont servis malgré `no-store`, **changer le port** dans `.claude/launch.json` (URLs neuves = cache froid). Après un `preview_resize`, la carte Leaflet garde sa taille (pas d'événement resize) → `map.invalidateSize()`, et les captures d'écran deviennent peu fiables → redémarrer le serveur de preview pour un onglet frais. La connexion Google (popup OAuth) n'est **pas** testable dans le preview (iframe sans session Google) : seul le site déployé fait foi. Enfin, le tampon de console du preview conserve parfois des erreurs d'anciens chargements — poser un `console.log` marqueur frais avant de conclure à un bug.
