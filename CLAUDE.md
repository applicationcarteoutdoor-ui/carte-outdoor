# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Projet

**SpotMap** (renommée en v62, ex-« Carte Outdoor » — le dépôt, l'URL GitHub Pages et les clés localStorage `carte-outdoor:*` NE CHANGENT PAS) : PWA statique de cartographie de points d'intérêt outdoor (via ferrata, refuges, escalade, canyons, châteaux, villages classés, randonnées, GR), **multi-pays** (France + Nouvelle-Zélande, page de garde = **vraie carte du monde cliquable** — tuiles OSM, zoom/déplacement LIBRES (minZoom 1), pays disponibles surlignés en teal semi-transparent + points verts pulsants, pays « bientôt » = points ambrés (liste `PAYS_BIENTOT` dans app.js), **nom du pays au survol** (tooltips ; `Intl.DisplayNames` fr sur `data/monde.geojson`, Natural Earth domaine public sans Antarctique — les polygones des autres pays sont invisibles mais survolables) ; cf. `js/config/pays.js` et `choisirPays()`). Logo : spot + « ? » (`icons/icon.svg`, PNG régénérés par `tools/generer_icones.py`). **Filtres VIERGES à chaque activation de catégorie** (le cochage d'une catégorie fait `delete state.filterSelections[id]` — demande utilisateur v64 : aucune sélection d'une session passée ne reste). **Vanilla JS + modules ES, aucun build, aucune dépendance npm** — les librairies (Leaflet, markercluster, togeojson, fflate) sont vendorées dans `vendor/`. Interface, commentaires et documentation en **français**. ~12 400 points par défaut (dont 3 351 sites d'escalade, 1 274 canyons, 1 174 cascades, 965 lacs, 810 châteaux, 380 villages et 135 randonnées sur 28 massifs) + 190 tracés GR + trois couches volumineuses en fichiers séparés chargés à la demande (non pré-cachés) : ~66 500 toilettes (`data/toilettes.geojson`), ~49 500 points d'eau (`data/eau.geojson` : fontaines, robinets, sources, avec filtre de potabilité) et ~49 700 grottes/cavités de spéléologie (`data/grottes.geojson` : socle Grottocenter ODbL, avec profondeur/développement/type/progression/eau et 6 filtres). Deux fonctions connectées : **l'Oracle** 🔮 (« quoi faire autour d'un code postal ou d'une commune » — mode gratuit sans clé, ou IA avec la clé Anthropic/Google de l'utilisateur) et la **sync multi-appareils** (Supabase + compte Google).

**En ligne** : https://applicationcarteoutdoor-ui.github.io/carte-outdoor/ (GitHub Pages, dépôt public `applicationcarteoutdoor-ui/carte-outdoor`, branche `main`). `Donné/` (sources perso, 117 Mo) et les caches de `tools/` sont exclus par `.gitignore`. **C'est l'utilisateur qui pousse (`git push`), jamais l'agent** : proposer le workflow, ne pas l'exécuter. Workflow de mise à jour : finaliser → bump `VERSION` dans `sw.js` → `git add -A && git commit && git push` → Pages redéploie (~1 min) → les visiteurs reçoivent la nouvelle version au chargement suivant. Publication Play Store (TWA/PWABuilder) : guide dans `docs/PLAYSTORE.md` ; `.well-known/assetlinks.json` attend encore l'empreinte SHA-256 de la Play Console. **`.nojekyll` est indispensable** : sans lui, Pages (Jekyll) ne sert pas les dossiers commençant par un point.

## Règles critiques

- **Incrémenter `VERSION` dans `sw.js` après toute modification de fichiers de l'app** — mais seulement quand l'ensemble est finalisé : une version bumpée en cours de chantier fige un état intermédiaire dans le cache des visiteurs (déjà causé des « bugs » fantômes). Depuis la v9, `app.js` recharge automatiquement la page au `controllerchange` : la nouvelle version s'applique au premier chargement. Depuis la v31, `app.js` appelle aussi `reg.update()` au retour au premier plan (`visibilitychange`) et toutes les heures → une PWA installée restée ouverte se met à jour seule.
- **Pré-cache du shell avec `fetch(new Request(url, { cache: "reload" }))`, jamais `cache.addAll()`** : `addAll` passe par le cache HTTP du navigateur, donc un fichier dont le CONTENU change sans que le NOM change (ex. `img/carnet-*.jpg` remplacées) était recopié PÉRIMÉ dans le nouveau cache — la VERSION montait mais les pixels restaient anciens (bug « couverture avec le bureau » v24→v26, plusieurs versions à diagnostiquer).
- **Jamais de `confirm()`/`alert()` natifs** : silencieusement ignorés dans certains contextes (PWA installée, dialogues bloqués) — utiliser `confirmer()` de `js/import-export.js` (dialogue intégré).
- **Échappement : uniquement `esc()` de `js/util.js`** (échappe aussi `"` et `'`, donc sûr dans les attributs). Jamais d'échappement local ni l'astuce `div.textContent` : elle n'échappe pas les guillemets → injection d'attribut possible depuis un fichier de sauvegarde importé (noms de points, libellés de catégories = données tierces).
- **CSP** : la balise meta de `index.html` liste les origines réseau autorisées — tout nouvel appel externe (fournisseur IA, back-end…) exige d'ajouter son domaine à `connect-src`, sinon blocage silencieux. Même piège pour les **images** : `img-src` doit lister `https://upload.wikimedia.org` (les photos des châteaux/cathédrales/cités viennent de Wikipédia — leur oubli les a fait disparaître de la v35 à la v41, sans aucune erreur visible).
- **Les clés API de l'Oracle sont secrètes** : localStorage de l'appareil uniquement, JAMAIS dans l'export de sauvegarde ni dans la sync. À l'inverse, l'URL et la clé `anon` Supabase (`js/config/supabase.js`) sont publiques par conception (RLS protège les données).
- **Ne jamais conseiller « vider les données du site »** à l'utilisateur : cela efface son carnet et ses points (IndexedDB) — et c'est inutile pour recevoir une mise à jour depuis les correctifs du SW.
- En dev, le service worker sert son pré-cache : pour tester une modification, désinscrire le SW + vider les caches dans le navigateur (ou mode navigation privée). `dev_server.py` envoie `no-store` mais ne neutralise pas le SW.
- Les **ids de catégorie** (`themes.js`) sont stables et stockés dans les données des points : ne jamais les renommer. Seul le `label` est modifiable. **Ids de points aussi stables** (statuts/carnet des utilisateurs pointent dessus) : jamais renumérotés ; une revue de données recale/enrichit EN PLACE, les ajouts prolongent la séquence.
- **Licences des données** : les **FAITS** (coordonnées, cotations, hauteurs, comptes) ne sont pas protégeables et sont réutilisables ; les **descriptions rédigées et les photos** relèvent du droit d'auteur/licences (CC-BY-SA de Camp to Camp, etc.) — **ne pas les recopier**, s'en tenir aux faits + un lien vers la source. Attribution requise pour Licence Ouverte/ODbL : section « Sources des données » dans le dialogue ⚙️ Réglages (`index.html`). Photos affichables : `upload.wikimedia.org` uniquement (CSP).
- `dev_server.py` doit rester sur `ThreadingHTTPServer` (un serveur mono-thread bloque le chargement : le navigateur ouvre des connexions parallèles).

## Architecture

**Multi-pays (v60)** : l'app couvre plusieurs pays — **`js/config/pays.js`** (SHELL) est le registre : par pays, fichier de points (`data/points.geojson` FR, `data/nz/points.geojson` NZ — **seul celui de la France est pré-caché**), surcouche de grands itinéraires (`gr.fichier/label/compte/fiche` : Sentiers GR ↔ Great Walks), catégories disponibles (`categories` = liste ou null=toutes, `categoriesExclues` — les catégories perso et celles avec points utilisateur restent toujours visibles), `couchesLourdes` (toilettes/eau/grottes = France seulement, FAB 🚻 masqué ailleurs), vue de départ et `wikiLang` (Oracle). Le pays choisi vit dans `carte-outdoor:pays` (localStorage, lecture SYNCHRONE — map.js/sidebar.js l'importent). **Page de garde** `#pays-overlay` au premier boot (`choisirPays()` dans app.js), bouton « 🌍 Changer de pays » dans ⚙️ Réglages (changement → `location.reload()`). Vue mémorisée **par pays** (prefs `vue_fr`/`vue_nz`). Au boot, les catégories actives inconnues du pays sont décochées (sinon leurs filtres polluent le panneau) ; si plus rien n'est coché, la 1re catégorie du pays est activée. **Le carnet et les statuts sont COMMUNS à tous les pays** (ids de points préfixés par pays : `nz-hut-0001`) : à l'ouverture du carnet, `chargerPointsAutresPays()` charge les points des autres pays (hook `avantOuverture` de carnet.js) pour résoudre tous les noms. Ajouter un pays = une entrée dans `PAYS` + `data/<pays>/points.geojson` (+ éventuelle surcouche d'itinéraires).

`js/app.js` est le seul propriétaire de l'état partagé (`state` : points, statuts, catégories cochées, sélections de filtres) et câble tous les modules par callbacks à l'init. Les modules ne s'importent presque pas entre eux ; le flux est : interaction → callback vers app.js → mutation de `state` → `rafraichir()` (re-rend panneau gauche + filtres + marqueurs + persiste les prefs). Les prefs sont écrites avec un debounce de 400 ms (`planifierPrefs`), vidé sur `visibilitychange`/`pagehide`. Au boot, un garde intercepte les retours OAuth : si `window.name === "carte-oauth-popup"` avec un jeton dans le hash, la page appelle `finaliserPopupAuth()` au lieu de démarrer l'app.

- **`js/config/themes.js`** — LE point de configuration : catégories (id stable, label, couleurs, icône), champs de fiche, et **filtres déclaratifs** (types `tokens`/`prefix`/`contains`/`value`/`bucket`, évalués par `passeFiltre()` de `js/filtrage.js` — fonction pure, couverte par `dev/tests.js` ; les buckets lisent les champs numériques suffixés `_n` des `details`). Trois couches de catégories : `THEMES` (base), `customThemes` (créées par l'utilisateur, enregistrées via `registerCustomThemes()` au boot, supprimables), `overrides` (personnalisation label/couleurs/icône, appliquée par `getTheme()`). Toujours passer par `getTheme()`/`allThemes()` pour l'affichage. Catégories récentes : `randonnee` (🥾, filtres par effort Durée/D+/Distance en buckets lisant `duree_n`/`denivele_n`/`distance_n` ; tracé au clic), `lac` (🏞️, filtres Fiche + Altitude en bucket), `cascade` et `chateau` (filtre Fiche). La catégorie `cite-caractere` porte le label unifié **« Village »** avec un filtre `contains` sur `label` (⭐ Plus beau village de France / 🏘️ Cité de caractère) — les deux anciennes catégories ont été fusionnées en v46 sous cet id stable.
- **Couches volumineuses (`toilettes`, `eau`, `grotte`)** — modèle commun dans app.js, centralisé par la table `COUCHES_LOURDES` : catégorie normale de `themes.js` mais dont les points vivent dans un fichier séparé NON pré-caché (`data/toilettes.geojson`, `data/eau.geojson`, `data/grottes.geojson`), chargé à la première activation (`chargerToilettes`/`chargerEau`/`chargerGrottes` → fusionnés dans `state.allPoints` par `chargerPoints`), rechargé au boot si la catégorie était déjà cochée. Le compteur affiche « … » tant que le fichier n'est pas chargé. À l'activation, un **dialogue d'avertissement** (`demanderModeCouche(dialogId)` sur `#wc-dialog`/`#eau-dialog`, boutons `.mode-retour/.mode-proximite/.mode-valider`) propose trois choix : Retour, « À moins de 1 km » (géoloc + zoom sur la position via `coucheAutourDeMoi(id)`, n'affiche que cette couche autour de soi), ou Valider (affichage complet). Ajouter une couche lourde = une entrée dans `COUCHES_LOURDES` + un `<dialog>` jumeau + un getter `pointsX`/`chargerX` + un compteur « … ». `grotte` : socle **Grottocenter** (API publique `api.grottocenter.org/api/v1/geoloc/entrances?sw_lat&…` — tout renvoyé par bbox en un appel ; FAITS ODbL seulement, jamais les topos/descriptions/photos protégés), `tools/recolter_grottes_grottocenter.py` + `tools/construire_grottes.py` (filtre France par région + géocodage inverse — la bbox mord sur l'Italie/Espagne ; type/progression/eau déduits du NOM ; fusion des 484 grottes Wikipédia `grot-XXXX` conservées + nouvelles `grot-g{id}` stables). Nombre de cordes / hauteur de puits = **impossibles en open data** (topos protégés). `eau` : récolté par `tools/recolter_eau.py` (Overpass `amenity=drinking_water`/`man_made=water_tap`/`natural=spring`…), `tools/normaliser_eau.py` pose `details.type` et `details.potabilite` en libellés lisibles (servant à la fois d'affichage et de valeur de filtre — la potabilité d'une source est « non garantie », jamais présumée). **Ne pas ajouter ces fichiers au SHELL de `sw.js`** (volumineux, mis en cache à la volée par le fetch handler).
- **`js/storage.js`** — toute la persistance, API volontairement async pour pouvoir brancher un back-end plus tard sans toucher au reste. localStorage (prefs, statuts, overrides, customThemes, idées) ; IndexedDB v3 (`DB_VERSION = 3`, stores : `userPoints`, `tracks`, `journal` = carnet par point avec photos, `carnet` = thème + images du grimoire dont le logo). Protection anti-effacement : `app.js` demande `navigator.storage.persist()` et rappelle une sauvegarde (export) au bout de 14 j sans export ; `dernierExport`/`dernierRappelSauvegarde` sont des prefs.
- **`js/map.js`** — performance : les marqueurs sont créés une fois et mis en cache (`cache: Map<id>`), les instances de divIcon sont partagées (une par combinaison thème/statut/sélection), l'icône n'est régénérée que si statut/thème/sélection change ; la **sélection fait partie de l'icône** (classe `pin-selected`) pour survivre aux reconstructions du clustering. Au-delà de 4 000 points, `setPoints` ajoute par tranches **triées par distance au centre** (la zone regardée s'affiche en ~1 s, le reste en fond) — **ne pas revenir au `chunkedLoading` de markercluster : il n'affiche RIEN avant la fin du lot complet** (~25 s pour 66 000 points). Attention : `map.getCenter()` pendant un `setView` animé renvoie l'ANCIEN centre (d'où `animate: false` dans `montrerRayon`). La carte est en `preferCanvas` : les polylignes (GR, traces) ne sont **pas** des `<path>` SVG — les tester via `map.eachLayer`, pas par sélecteur CSS. Une polyligne canvas n'est cliquable qu'au pixel près : les GR utilisent donc un renderer dédié `L.canvas({ tolerance: 12 })` (zone de clic élargie) — sans lui, impossible de toucher le tracé (bug corrigé en v47). Les GR sont rouges pointillés (`GR_STYLE`), orange plein une fois sélectionnés (`GR_STYLE_ACTIF`). **Itinéraire épinglé (randonnée, canyon OU sentier GR)** — même modèle d'interaction, `itineraireEpingle = { type: "rando"|"gr"|"canyon", … }` dans app.js : au clic, une fiche s'ouvre, le tracé **reste affiché après fermeture** (pastille `#rando-chip` en haut-droit pour rouvrir la fiche), **un seul épinglé à la fois** (ouvrir l'autre remplace le précédent), décocher la catégorie/surcouche efface tout. Randonnée : `montrerTraceRando`/`cacherTraceRando`/`getTraceRando` dessinent le tracé de `data/randos.geojson` (vert `TRACE_RANDO_STYLE`, `interactive:false`) ; fiche via `openDetails`. **Canyon** : même mécanisme **généralisé** (`chargerTraces(fichier)` en cache par fichier, calque unique partagé `traceActive`) — `montrerTraceCanyon`/`getTraceCanyon` lisent `data/canyons-traces.geojson` (MultiLineString reconstruite du cours d'eau OSM, teal pointillé `TRACE_CANYON_STYLE`) ; seuls ~27 canyons ont un tracé (propriété `properties.trace` du point → affiche le bouton GPX). GR : `selectionnerGr(grId)`/`deselectionnerGr()` surlignent la couche existante (id de session `grId = gr-<index>` posé au chargement), clic → callback `onGrClick` → `openDetailsGr` (fiche allégée : nom, distance, D+ estimé, liens, pas de statut/carnet). **GPX** pour les deux (bouton 📥 GPX) : `telechargerGpx(nom, segments, depart?)` d'app.js construit le fichier (Blob + `a[download]`) — bouton de fiche aiguillé par `telechargerGpxItineraire` (rando via `getTraceRando`, canyon via `getTraceCanyon` en aplatissant la MultiLineString en segments), GR via sa géométrie LineString directe. Point GPS : `toggleLocate` pose un `L.divIcon` `.position-gps` (gros point + halo pulsant `.position-gps-halo`). Surcouches chargées à la demande : `data/gr.geojson` (2 Mo), `data/toilettes.geojson` (14 Mo), `data/randos.geojson` (205 Ko, tracés des 135 randos), `data/canyons-traces.geojson` (27 tracés de canyons). Fonds de carte : `getFonds()`/`setFond()` exportés (le `L.control.layers` natif a été retiré au profit du dialogue `#layers-dialog`).
- **`js/import-export.js`** — import unifié (un seul `<input>` `#file-import`) : tout est converti en GeoJSON (togeojson pour GPX/KML, fflate pour KMZ), puis routé — LineStrings → traces (`gpx.js`), Points → dialogue de validation (erreurs listées par ligne, choix de catégorie si absente). L'export `formatVersion: 2` embarque points + statuts + carnet + customThemes. Formats documentés dans `docs/FORMAT-IMPORT.md`.
- **`js/carnet.js`** — « le Carnet », grimoire de sorties : couverture + pages en images de fond (`img/carnet-couverture.jpg`, `img/carnet-page-1..4.jpg`, pré-cachées), historique manuscrit par lieu. Personnalisable via un dialogue de thème (grimoire / voyage / nuit) et un **logo utilisateur** rond incrusté sur la plaque de couverture (`.couv-logo`, positionné en % pour masquer le symbole « ? » de l'image). Réglages persistés dans le store IndexedDB `carnet`.
- **`js/oracle.js`** + **`css/oracle.css`** — l'Oracle : depuis un code postal, un **nom de commune/village** (`resoudreNomCommune`, geo.api.gouv.fr `?nom=&boost=population`) ou le GPS, révèle quoi faire autour. Les points annoncés sont des **liens cliquables** (`{{p:ID}}…{{/p}}` → bouton `.oracle-point`, id validé `[\w-]+`) qui recentrent la carte sur le lieu. Deux modes : **libre** (gratuit, par défaut sans clé — points de la propre carte à ≤ 40 km avec distance/direction, Wikipédia `list=geosearch`, liens d'agendas) et **IA** (abstraction `FOURNISSEURS` : Anthropic — en-tête `anthropic-dangerous-direct-browser-access` + outil `web_search` — et Google Gemini + grounding `google_search`). **Ne pas réintroduire OpenAI** : son API n'envoie pas d'en-têtes CORS, injoignable depuis un navigateur (vérifié). La sortie du modèle passe par un parseur strict (`##` sections, `-` puces, `**gras**`) + `esc()` ; URLs de sources validées `^https?://`. UI boule de cristal : **sphère** au repos (accueil, chargement, erreur) qui se déploie en **capsule rectangulaire de verre** (`forme-etiree`, coins ~26px) pour la lecture des résultats et le panneau des clés — choix utilisateur v41, ne pas revenir à l'ellipse.
- **`js/sync.js`** — sync multi-appareils Supabase **sans SDK** (fetch pur ; config `js/config/supabase.js`, guide `docs/SYNC.md`). Connexion Google en **popup** : `window.open(url, "carte-oauth-popup")` — le nom de fenêtre survit aux redirections OAuth, le popup écrit la session en localStorage et la fenêtre principale la capte via l'événement `storage` (repli : redirection pleine page si popup bloqué). Flux implicite (`#access_token`). **Rafraîchissement du jeton sérialisé** (une seule promesse partagée) : Supabase fait tourner les refresh tokens, deux refresh concurrents déconnectent. Données dans une table clé/valeur `donnees` sous RLS ; tirer → fusionner via les fonctions de fusion de `storage.js` (jamais de perte) → pousser ; rechargement de la page si le distant apporte du neuf.
- **`js/config/glossaire.js`** — définitions des acronymes (cotations…) ; `glossaireHTML()` échappe et enveloppe automatiquement les termes connus d'une infobulle. Une entrée à ajouter suffit pour un nouvel acronyme.
- **`js/communaute.js`** — catégories COMMUNAUTAIRES (v63) : partager ses catégories perso / importer celles des autres, avec **validation manuelle** avant publication. Serveur = Supabase table `categories_partagees` (schéma `supabase/communaute-schema.sql` : RLS force auteur+statut en_attente à l'insert, lecture publique des seules `validee`, modération via table `moderateurs` sans policy, RPC `compter_telechargement` security definer). Colis ≤ 500 points/1 Mo, **sans photos** (droits+poids) ; import re-identifié (`comm-<id8>`, ids de points stables → réimport idempotent), coordonnées/liens/champs contrôlés, affichage via `esc()`. Session/connexion Google réutilisées de sync.js (`sessionCommunaute()`/`demanderConnexion()` exportées). Dialogue `#communaute-dialog` (volets Importer/Partager), boutons page de garde + ⚙️ Réglages. Tant que le SQL n'est pas exécuté → messages doux (« pas encore ouverte »). Guide de modération : `docs/COMMUNAUTE.md`.
- **`js/partage.js`** — partage de l'app : bouton flottant (sous 🏃 position ; icône SVG de partage universelle « trois nœuds reliés », **surtout pas un QR** — retour utilisateur) → dialogue `#partage-dialog` avec QR en grand (`img/qr-site.svg`, SVG statique généré par `tools/generer_qr.py` + segno, plaque blanche imposée pour rester scannable en mode nuit), bouton partage natif (`navigator.share`, masqué si non géré) et copie du lien. L'URL du site est en dur dans le module ET dans le QR : si le site déménage, relancer `generer_qr.py`.

## Pipeline de données (`tools/`)

`build_data.py` + `enrichissements.py` génèrent `data/points.geojson` et `data/gr.geojson` depuis `Donné/` et des sources en ligne. **Tout est mis en cache dans `tools/`** (`communes.json`, `refuges-api.json`, `wiki-cache.json`, `alti-cache.json`, `viaferrata-liste.html`, `grgo-index.html`) : relancer le script ne re-télécharge que le manquant.

- Refuges : API refuges.info (`api/bbox…detail=complet`) — le KML de `Donné/Refuge` est obsolète.
- Escalade/châteaux/cités : géocodés à l'origine au **centroïde de commune** (référentiel geo.api.gouv.fr, rapprochement exact → préfixe → contient, St↔Saint normalisé) — d'où la nécessité des **revues de données** (ci-dessous) pour recaler sur les vraies positions.
- Wikipédia : **ne pas utiliser l'endpoint de recherche** (rate-limité en rafale et renvoie l'article de la commune) — construire des titres candidats (« Château de X »…) et les vérifier par lots de 50 via `action=query&titles=`, en sauvegardant le cache à chaque tranche (les 429 arrivent par vagues, le retry attend 15-180 s). Les châteaux sans page trouvée restent dans `chateau` avec `details.fiche = "À vérifier"` (l'ancienne catégorie séparée `chateau-a-verifier` a été fusionnée en v45 — le filtre déclaratif « Fiche » ✅ Référencé / 🔍 À vérifier la remplace ; même convention pour les cascades et toute nouvelle catégorie).
- Grottes et cathédrales : récolte des arbres de catégories Wikipédia (`recolter_categorie_wikipedia`, sous-catégories suivies seulement si leur nom contient le mot-clé, sinon le parcours dérive hors sujet) puis `prop=coordinates|pageimages` — **`colimit=max` et suivi de `continue` obligatoires**, sinon l'API ne renvoie que 10 coordonnées par réponse (bug déjà rencontré : 80 % des points silencieusement sans coordonnées).
- D+ des GR : service altimétrique IGN (`data.geopf.fr/altimetrie`, POST par lots de 180 points, échantillonnage ~300 m) — c'est une estimation, affichée comme telle.
- Cascades / lacs / villages : récoltés par l'agent enrichisseur (scripts `recolter_cascades.py`, `recolter_lacs.py`, `recolter_villages.py`), avec **sélection qualitative** (les lacs anonymes ou minuscules sont écartés — 26 % de rejet) et convention `details.fiche`. Lacs recentrés dans l'eau, altitude corrigée par l'IGN.
- Randonnées : `recolter_randonnees.py` (points de départ + sommets, champ `massif`) puis `recolter_traces_randos.py` calcule le vrai **tracé sur le réseau de sentiers OSM** (routage Dijkstra départ→sommet, via ferrata/privé exclus, simplification Douglas-Peucker) → `data/randos.geojson`. Ensuite **`completer_stats_randos.py` est OBLIGATOIRE** : distance mesurée sur le tracé (détection aller/boucle), dénivelé IGN si manquant, durée estimée (4 km/h + 300 m/h). Il pose les champs numériques `distance_n`/`denivele_n`/`duree_n` qui alimentent les filtres par effort de la fiche randonnée. Chaque fiche porte 3 liens externes (`properties.links` : 🥾 Komoot, ⛰️ Altitude Rando, 🗺️ AllTrails). **Couverture France livrée : 135 randonnées, 28 massifs** (listes éditoriales `tools/randos_liste_*.py` + `recolter_randonnees_france.py` ; cache des tracés shardé `tools/traces-cache/`). Limite honnête : le tracé est le plus direct sur sentiers balisés, pas les variantes en boucle éditoriales.
- Toilettes : API Overpass (`amenity=toilets`, tuiles 3°, nœuds + centres de bâtiments, `access=private/no` écartés) → `data/toilettes.geojson` séparé. Overpass rate-limite fort (429/504) : retries longs, cache sauvegardé seulement si toutes les tuiles réussissent. Le GeoJSON « toilets » fourni dans `Donné/` ne contient QUE des restaurants (mauvais export) — ne pas l'utiliser.
- **Nouvelle-Zélande** (`data/nz/`, 3 930 points sur 9 catégories + 11 Great Walks) : huttes/campings **DOC** (ArcGIS `*_DTO` — les datasets « Deprecated » répondent encore, ne PAS les utiliser ; `outSR=4326` OBLIGATOIRE (natif NZTM) ; pagination `resultOffset` SANS `orderByFields` = instable (doublons) ; champs métier en paires pivot `CharName#/CharValue#` au mapping variable → pivoter par NOM) + lacs/cascades/grottes **NZ Gazetteer** (`gaz.csv` sans clé, BOM UTF-8, filtrer lat -34/-48,5 sinon Antarctique) + compléments **OSM** (`area ISO3166-1=NZ`, JAMAIS de bbox : les Chatham chevauchent l'antiméridien) + grottes/cathédrales **Wikipédia EN** (photos Commons, `recolter_nz_wikipedia.py` — mêmes pièges que la France : colimit=max + continue, sous-catégories par mot-clé). Great Walks : segments EAM par **préfixe FlocID** (filtrer sur `SubObjectType='Great Walk'` TROUE les tracés — les accès sont « Walking Track » ; exception LHAUROBC restreint au type Great Walk sinon Hump Ridge avale le South Coast Track). Listes ÉDITORIALES vérifiées (dans `construire_nz.py` + `tools/nz-villages.json`) : 7 châteaux/forts (les 75 « homesteads » OSM écartés), 22 villages de caractère (aucun label officiel en NZ), 3 via ferrata (la « High Canopy » du zoo d'Auckland est un habitat d'animaux tagué via_ferrata — écartée), **26 randonnées emblématiques** (`recolter_nz_randos.py` : FlocID EXACTS vérifiés à la main — Roys Peak, Mueller Hut, Rob Roy, Avalanche Peak, Hollyford… — tracés réels DOC dans `data/nz/randos.geojson` (non pré-caché, `pays.fichierRandos` aiguille map.js), distance mesurée, durée estimée 4 km/h, PAS de D+). Escalade NZ : pas de source libre (ClimbNZ fermé), abandonnée. Scripts `recolter_nz_doc.py` + `recolter_nz_gazetteer.py` + `recolter_nz_osm.py` + `recolter_nz_wikipedia.py` + `construire_nz.py` (ids stables `nz-hut/camp/lac/casc/grot/cath/chat/vill/vf-####`). Licences : DOC CC-BY 4.0, LINZ CC-BY 4.0, OSM ODbL. **`grotte` en NZ = catégorie normale** (~130 points dans points.geojson) : le mécanisme couche lourde est conditionné au pays via `coucheLourde(id)` dans app.js.
- Canyon (canyonisme) : socle **RES / Data ES** (`equip_type_name="Canyon"`, **Licence Ouverte** — `tools/recolter_canyon_res.py`, 1 274 sites : position, `equip_long`=longueur, `equip_haut`=dénivelé, DOM inclus) + faits **OSM** (`relation[route=canyoning]`, ODbL — `tools/recolter_canyon_osm.py` : cotation FFME, `max:abseil`→corde estimée, site web, **tracé** reconstruit des ways membres en MultiLineString) ; fusion `tools/fusion_canyon.py` (rapprochement OSM→RES conservateur proximité+nom, ~27/32, ids stables `canyon-####`, écrit points.geojson + `data/canyons-traces.geojson`). **Camp to Camp n'a PAS le canyoning** (filtre API ignoré). **descente-canyon.com écarté** : CC BY-NC-SA + droit du producteur de base de données (clause non-commercial incompatible store, extraction massive risquée) — on ne le scrape jamais, lien seulement. **Temps d'approche/retour = impossibles en open data** (topos protégés), comme le nb de cordes en spéléo.
- Culture : OSM (musées/galeries/sites archéo/monuments, name obligatoire) via `recolter_culture_osm.py` (FR ~10 min Overpass) → `construire_culture.py` (FR = couche lourde `data/culture.geojson`, NZ = merge points). Enrichissement `enrichir_culture_wikipedia.py` (musées+galeries seulement — monuments = homonymes ; nom EXACT + coords article ≤ 10 km, photos+lien+description 1re phrase, cache par lot). **PIÈGE Wikimedia (2025+) : les vignettes n'acceptent QUE des tailles normalisées** (500, 960, 1280 OK ; 640/800/900/1024 → HTTP 400 « Use thumbnail sizes listed ») — l'API pageimages normalise d'elle-même (pithumbsize=900 → URL 960px), mais toute construction MANUELLE d'URL `/NNNpx-` doit s'en tenir à la liste ; `agrandir_photos.py` (500→960, HEAD-vérifié : Wikimedia refuse d'agrandir au-delà de l'original) documente la méthode sûre.
- Les échecs (lieux-dits, communes fusionnées, ~420 sites) sont listés en fin d'exécution ; c'est une limite des données sources, pas un bug.

## Revue des données (catégorie par catégorie)

Chantier en cours (`dev/ROADMAP.md` étape 1) : vérifier position/infos et compléter chaque catégorie en croisant des sources fiables. Méthode validée (via-ferrata, escalade), réutilisable pour cascade/château/lac :

2. **Licences d'abord** (workflow de recherche multi-sources si besoin) : identifier une source ouverte réutilisable. Ordre de préférence : open data officiel (data.gouv/Etalab **Licence Ouverte**) > bases spécialisées (OSM ODbL, Camp to Camp) — **faits seulement**, jamais prose/photos.
3. **Recalage GPS conservateur** : recaler un point sur une source SEULEMENT si la correspondance est certaine (nom exact/⊂, ou proximité unique, ou même commune) — sinon laisser, jamais deviner (recaler sur le mauvais site est pire).

## Pièges connus de l'environnement de preview

**Si le panneau de preview intégré est indisponible** (arrive selon les sessions) : piloter Chrome headless via CDP comme `dev/video/capturer.py` (classe `Cdp` réutilisable, `Runtime.evaluate` avec `awaitPromise`/`returnByValue`, `Emulation.setDeviceMetricsOverride` pour le mobile) sur le serveur de dev. Toujours **purger SW + caches + `localStorage`** avant de tester (sinon ancienne version / prefs polluées) ; les catégories de la sidebar sont des `.cat-row` (pas des `<li>`) ; les transitions CSS restent parfois figées en émulation mobile (tester avec `transition:none` avant de crier au bug). Le proxy du panneau de preview cache agressivement : si des fichiers périmés sont servis malgré `no-store`, **changer le port** dans `.claude/launch.json` (URLs neuves = cache froid). Après un `preview_resize`, la carte Leaflet garde sa taille (pas d'événement resize) → `map.invalidateSize()`, et les captures d'écran deviennent peu fiables → redémarrer le serveur de preview pour un onglet frais. La connexion Google (popup OAuth) n'est **pas** testable dans le preview (iframe sans session Google) : seul le site déployé fait foi. Enfin, le tampon de console du preview conserve parfois des erreurs d'anciens chargements — poser un `console.log` marqueur frais avant de conclure à un bug.
